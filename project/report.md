## 2. Постановка задачи и контекст

**Тип задачи:** Рекомендация элементов (item recommendation), регрессия (предсказание рейтингов)

**Входные данные:**
- User-Item матрица: оценки пользователей (userId, movieId, rating)
- Метаинформация о фильмах: название, жанры, год выпуска
- Дополнительно: временные метки, теги

**Целевой выход (target):**
- Для задачи **recommendation**: упорядоченный список индексов фильмов (top-k), не оценённых пользователем
- Для задачи **rating prediction**: скалярное значение предсказанного рейтинга (0.5 - 5.0)

**Ограничения:**
- **Скорость:** API должен отвечать за <1 сек (для interactive рекомендаций)
- **Уникальность рекомендаций:** часто повторяются рекомендации для различных пользователей, поскольку данные не обновляются 
- **Quality:** Избегать пустых / неактуальных рекомендаций

### 2.3. Целевые метрики качества

Для оценки качества рекомендаций используем **offline metrics** (не с реальными пользователями):

| Метрика | Назначение | Диапазон |
|---------|-----------|----------|
| **NDCG@10** (Normalized Discounted Cumulative Gain) | Учитывает позицию релевантного элемента в top-10; важен порядок | [0, 1] |
| **Recall@10** | Доля релевантных фильмов, найденных в top-10 | [0, 1] |
| **Precision@10** | Доля релевантных фильмов среди top-10 | [0, 1] |
| **RMSE** (Root Mean Squared Error) | Ошибка предсказания рейтинга | ≥ 0 |
| **MAE** (Mean Absolute Error) | Средняя абсолютная ошибка оценки | ≥ 0 |

**Почему эти метрики:**
- NDCG, Recall, Precision — стандартные для ранжирования и рекомендаций
- Учитывают не только наличие релевантного элемента, но и его позицию
- RMSE/MAE — для оценки предсказания конкретных рейтингов (важно для гибридных подходов)
- Релевантным считаем фильм, если пользователь дал ему оценку ≥ 3.5 из 5.0

## 3. Данные

### 3.1. Источник данных

**Датасет:** MovieLens 100K (open-source)  
**Лицензия:** GroupLens Research  
**Ссылка:** https://grouplens.org/datasets/movielens/

Выбран open-source датасет, так как:

- Все данные обезличены (нет персональной информации)
- Есть метаинформация (жанры, года)


### 3.2. Структура данных

**Основные таблицы:**

| Таблица | Строк | Колонки | Описание |
|---------|-------|---------|---------|
| `ratings.csv` | 100,836 | userId, movieId, rating, timestamp | Основная таблица оценок |
| `movies.csv` | 9,742 | movieId, title, genres | Метаинформация о фильмах |
| `tags.csv` | 3,683 | userId, movieId, tag, timestamp | Пользовательские теги |
| `links.csv` | 9,742 | movieId, imdbId, tmdbId | Ссылки на внешние БД |

**Ключевые характеристики (из EDA):**

```
Рейтинги:
  - Диапазон: 0.5 - 5.0 (полшага)
  - Среднее: ≈3.5
  - Распределение: скошено влево (люди ставят больше высоких оценок)
  
Пользователи:
  - Всего: 610
  - Средне оценок на пользователя: 165 (wide range: 1-2698)
  - "Long tail": многие пользователи оценили <20 фильмов
  
Фильмы:
  - Всего: 9,742
  - Диапазон лет: 1888 - 2018
  - Жанры: 20 категорий (многие фильмы имеют несколько жанров)
  - Популярность: скошено (несколько фильмов имеют 100+ оценок, большинство <10)
```

### 3.3. Предобработка и EDA

**Выполненные шаги:**

1. **Очистка данных** (ноутбук: `notebooks/eda.ipynb`):
   - Удалены дубликаты
   - Проверены пропуски (их почти нет)
   - Валидация диапазонов значений (рейтинги 0.5-5.0)

2. **Feature Engineering**:
   - Извлечение года выпуска из названия фильма (regex)
   - Группировка по "эрам" (Old, 1980s, 1990s, 2000s, 2010s+)
   - **One-hot encoding жанров**: из строки "Action|Adventure|Sci-Fi" → 20 бинарных признаков
   - Временные признаки: month, weekday, is_weekend (из timestamp)

3. **Сохранение предобработанных данных**:
   - Объединённый датасет с one-hot жанрами сохранён в `src/data/preprocessed_data/full_data.parquet`

4. **Важные наблюдения (EDA insights)**:

   - **Cold start problem**: ~120 пользователей оценили <5 фильмов
     → Для них коллаборативная фильтрация плохо работает, нужен fallback
   
   - **Популярные фильмы смещены**: классические фильмы (70-90е) переrepresentированы
     → Может привести к смещению рекомендаций
   
   - **Жанровые корреляции**: Action и Adventure часто идут вместе
     → Гибридный подход (SVD + жанры) может помочь

**Ноутбук EDA:** `notebooks/eda.ipynb` — графики распределений, статистика по временам, анализ популярности

## 4. Модели и подходы

1. **Базовые (baseline) модели:**
  - В качестве baseline использовались простые подходы:
  - случайные рекомендации;
  - рекомендации по популярности (фильмы с большим числом оценок и высоким средним рейтингом);
  - предсказание рейтинга через глобальное среднее или случайное значение.
- Результаты:
  - ranking-метрики очень низкие(0.006-0.008);
  - RMSE высокий;
  - baseline показал необходимость модели, учитывающей историю пользователя и сходство.
- Ноутбук: `notebooks/baseline_model.ipynb`, `notebooks/item_based_collaborative_filtering.ipynb`
2. **Улучшенные модели и эксперименты:**
   - User-based CF (сходство пользователей по cosine); Item-based CF (сходство фильмов по cosine);
   - число соседей (k); порог релевантности для рекомендаций (rating ≥ 3.5); способ заполнения пропусков и нормализация рейтингов; фильтрация редких пользователей/фильмов;
   - сравнение User CF и Item CF;  тестирование fallback на популярные фильмы для холодного старта.
- Ноутбук: `notebooks/item_based_collaborative_filtering.ipynb`
3. **Нейросетевые модели (если применимо):**
   - SVD с эмбеддингами пользователей и фильмов; 
   - Hybrid SVD с учётом жанров через one-hot encoding; 
   - user embedding + item embedding; user bias + item bias + global bias; для Hybrid SVD — дополнительный genre layer;
   - оптимизатор Adam; 
   - learning rate ~0.005; 
   - batch size ~256; \
   - loss = RMSE; 
   - EarlyStopping(patience=5).
- Ноутбук: `notebooks/svd_and_hybrid_svd.ipynb`

## 5. Экспериментальный протокол и результаты

Здесь важно показать, что выбор финальной модели был осмысленным.

1. **Экспериментальный протокол:**
   -  train / val / test(0.7; 0.15; 0.15);
   - на валидационной выборке.
   - Метрики:
      - ranking: Precision@10, Recall@10, NDCG@10;
      - regression: RMSE, MAE.
   - Оценка:
      - ranking считался на top-10 рекомендаций;
      - RMSE/MAE на валидационной выборке для предсказаний рейтингов.
2. **Сравнение моделей по метрикам:**

  | Модель | Описание | NDCG@10 | Recall@10 | Precision@10 | RMSE | Комментарий |
|---|---|---|---|---|---|---|
| Baseline | случайные / популярные рекомендации | 0.0017 | 0.0012 | 0.0014 |  1.63 | стартовая точка |
| User CF | cosine user-user | 0.157 | 0.14 | 0.071 | 0.97 | основная модель |
| Item CF | cosine item-item | 0.105 | 0.126 | 0.071 | 1.11 | альтернативный подход |
| SVD | эмбеддинги пользователей/фильмов | 0.05 | 0.026 | 0.05 | 0.88 | нейросетевая модель |
| Hybrid SVD | SVD + жанры | 0.005 | 0.002 | 0.006 | 0.93 | ожидалась, что будет лучшей         |

3. Выбор финальной модели
- Финальная модель: User-based CF.
- Причины:
  - хороший баланс качества и простоты;
  - невысокие требования к ресурсам;
  - понятная интерпретация;
  - стабильность работы.
- Trade-off:
  - Hybrid SVD ожидалась лучшей моделью, поскольку учитывает контекстный признак в виде жанра фильма, однако коллаборативная модель показала лучший результат;
  - User CF предпочтительнее для рабочего REST-сервиса.

## 6. Архитектура решения и сервис

### 6.1. Архитектура пайплайна
- Данные загружаются из `src/data/`.
- В ноутбуках выполняется предобработка и обучение.
- Сохранение модели в `src/models/`.
- Сервис `src/service/main.py` загружает модель и обслуживает запросы.
- данные → предобработка → обучение → сохранение модели → сервис → API-запросы.

### 6.2. API и endpoints
- Реализовано:
  - `GET /health` — проверка статуса сервиса;
  - `GET /recommend/{user_id}` — выдача топ-k рекомендаций;
  - `GET /` — информация о сервисе.
- `/recommend` возвращает `user_id`, список рекомендаций и `count`.

### 6.3. Технологический стек
- Python 3.10+;
- FastAPI;
- Uvicorn;
- pandas, numpy;
- scikit-learn;
- PyTorch (для экспериментов);
- pickle для сериализации моделей.
- Запуск:
  - `uvicorn src.service.main:app --reload --host 127.0.0.1 --port 8000`
